三大陷阱（必读）
================

陷阱 1: 磁盘缩容问题 ⚠️ [致命]
  描述: 模板 20G，你要 8G，Proxmox 无法缩容
  后果: Terraform 卸载原盘，创建新空盘，VM 无法启动
  防御: 在脚本中验证 disk_size >= template_size
  来源: 2025-12-04 学习笔记

陷阱 2: Output 穿透问题 ⚠️ [常见]
  描述: 模块定义的 Output 不会自动对外暴露
  后果: 看不到 VM IP、VM ID 等关键信息
  防御: 在 Root Module 中显式定义 Output
  来源: 2025-11-28, 2025-11-30 学习笔记

陷阱 3: SSH Key 维护陷阱 ⚠️ [运维]
  描述: 频繁修改 Terraform 中的 sshkeys 触发 VM 重建
  后果: 服务中断，用户被迫切换 Key
  防御: 使用 Bootstrap Key 模式，日常由 Ansible 维护
  来源: 2026-01-28 学习笔记

五个核心命令
============

必须掌握（顺序很重要）:
  1. terraform init      # 初始化，下载 Provider
  2. terraform plan      # 查看执行计划（最重要！）
  3. terraform apply     # 应用配置
  4. terraform state     # 管理状态文件
  5. terraform destroy   # 销毁资源

记住顺序的关键：
  "初始化、规划、应用、管理、销毁"

为什么 plan 最重要：
  计划显示了即将执行的所有操作
  显示中断、等待时间长、意外重建等都能提前发现
  在 apply 前务必仔细检查 plan 输出

七个最佳实践
============

1. 显式优于隐式
   不要依赖默认值，显式定义所有关键参数
   示例: boot = "order=scsi0;ide2;net0"

2. 使用远程状态后端
   团队协作必须使用 S3/Terraform Cloud/Consul
   启用加密和版本控制

3. 模块化优先
   从第一个 VM 开始就设计模块
   支持代码复用和多环境部署

4. 严格的状态漂移检测
   定期运行 terraform plan
   任何非预期变更立即调查原因

5. 使用 lifecycle 块精细控制
   ignore_changes 忽略无关漂移
   prevent_destroy 保护关键资源

6. SSH 密钥与配置管理分离
   Terraform: Bootstrap Key（长期不变）
   Ansible: User Management（日常维护）

7. 充分的脚本验证
   生成 Terraform 变量的脚本需要验证单位、约束
   防止磁盘缩容、单位错误等致命问题
